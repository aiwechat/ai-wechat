# 协议说明

本文档定义客户端和服务端之间的应用层协议。项目底层使用 TCP socket 传输，应用层使用“4 字节长度前缀 + JSON Envelope”的格式来表达一条完整聊天消息。

这份协议不是在重新实现 TCP，也不是在伪造 TCP header。真正的 TCP header 由操作系统自动添加和解析。本项目自己定义的是 **TCP payload 里面的应用层消息格式**。

## 1. 为什么需要自己设计应用层协议

Python 的 `socket.sendall()` 和 `socket.recv()` 只能发送和接收 `bytes`。

也就是说，下面这些对象都不能直接在网络上传输：

```python
{"type": "private_msg", "content": "hello"}
ProtocolMessage(...)
```

它们必须先被转换成字节：

```text
Python 对象
-> dict
-> JSON 字符串
-> UTF-8 bytes
-> socket.sendall(bytes)
```

TCP 只保证这些字节会可靠、有序地到达对方，但 TCP 不知道“哪几个字节是一条聊天消息”。这就是原生 TCP socket 编程必须解决的问题。

例如客户端连续发送两条消息：

```text
消息1: {"type":"login"}
消息2: {"type":"private_msg"}
```

服务端调用 `recv()` 时，不一定刚好收到一条完整消息。可能出现：

```text
recv() -> {"type":"login"}{"type":"private_msg"}
```

这叫粘包。

也可能出现：

```text
recv() -> {"type":"priv
recv() -> ate_msg"}
```

这叫拆包或半包。

因此，本项目在 TCP 之上定义了一个应用层分帧规则：

```text
[4 字节 JSON 长度][JSON 消息体]
```

这 4 字节不是 TCP header，而是我们聊天协议自己的 frame header，用来告诉程序：后面多少字节才是一条完整 JSON 消息。

## 2. 和 HTTP / FastAPI 的类比

如果使用 FastAPI，代码看起来不需要自己处理这些细节：

```python
@app.post("/login")
def login(data: LoginRequest):
    ...
```

原因不是 FastAPI 不需要协议，而是 FastAPI 使用了 HTTP 这个现成的应用层协议。HTTP 已经定义好了：

- 请求从哪里开始。
- 请求头有哪些字段。
- Body 有多长。
- Body 是 JSON、表单还是文件。
- 路由如何表达。
- 错误如何表达。

一个 HTTP 请求本质上也是跑在 TCP 上的，真实格式类似：

```http
POST /login HTTP/1.1
Host: localhost:8000
Content-Type: application/json
Content-Length: 42

{"username":"alice","password":"123456"}
```

注意这里的：

```http
Content-Length: 42
```

它的作用和本项目的 4 字节长度前缀很像：告诉接收方 body 有多长。

区别是：

| 问题 | FastAPI / HTTP | 本项目原生 TCP |
| --- | --- | --- |
| 底层传输 | TCP | TCP |
| 应用层协议 | HTTP 已经定义 | 我们自己定义 |
| 消息长度 | `Content-Length` 或 chunked | 4 字节 big-endian 长度 |
| 业务路由 | URL + method，如 `POST /login` | JSON 里的 `type` 字段 |
| 业务数据 | HTTP body，常用 JSON | JSON Envelope 的 `payload` |
| 错误表达 | HTTP status code + response body | `type = "error"` 的 JSON 消息 |
| 框架支持 | FastAPI/Uvicorn 自动解析 | `common/protocol.py` 负责解析 |

所以可以这样理解：

```text
FastAPI 项目：
业务代码 -> FastAPI -> HTTP -> TCP -> IP

本项目：
业务代码 -> 自定义聊天协议 -> TCP -> IP
```

本项目选择原生 TCP，是为了体现计算机网络课程中的 TCP 长连接、应用层分帧、粘包/拆包处理、心跳检测、消息路由和并发连接管理。

## 3. 真实传输过程中到底发生了什么

以 Alice 给 Bob 发私聊为例，Python 代码中先构造消息：

```python
from common.protocol import MessageType, ProtocolMessage, encode_frame

message = ProtocolMessage(
    type=MessageType.PRIVATE_MSG,
    sender="alice",
    receiver="bob",
    payload={"content": "hello"},
)

frame = encode_frame(message)
sock.sendall(frame)
```

整个过程如下：

```text
1. ProtocolMessage Python 对象
2. message.to_dict()
3. json.dumps(...)
4. UTF-8 编码成 bytes
5. encode_frame(...) 在前面加 4 字节长度
6. socket.sendall(frame)
7. 操作系统自动添加 TCP header、IP header 并发送
8. 对方操作系统收到数据，去掉 TCP/IP header
9. Python socket.recv(...) 得到 TCP payload bytes
10. decode_frames(...) 根据前 4 字节拆出完整 JSON
11. json.loads(...) 转成 dict
12. ProtocolMessage.from_dict(...) 转回 Python 对象
13. 服务端根据 message.type 路由处理
```

程序自己能看到和处理的是：

```text
[4 字节长度][JSON UTF-8 bytes]
```

真正的网络报文大致是：

```text
[IP header][TCP header][应用层 payload]
                      |
                      +-- [4 字节长度][JSON body]
```

TCP header 的作用由操作系统负责，包括端口、序列号、确认号、校验和、窗口大小、SYN/ACK/FIN 等。项目代码不直接读写 TCP header。

## 4. 帧格式

每个 TCP payload 中的聊天 frame 由两部分组成：

```text
0                   4                   4 + body_length
+-------------------+-------------------+
| body_length (u32) | UTF-8 JSON body   |
+-------------------+-------------------+
```

- `body_length`：4 字节无符号整数，big-endian，表示 JSON body 的字节数。
- `JSON body`：UTF-8 编码的 JSON Envelope。
- 当前最大帧大小为 8 MB。

例如：

```text
b'\x00\x00\x00\x52{"version":"1.0","type":"heartbeat","payload":{"seq":1},...}'
```

前 4 字节 `\x00\x00\x00\x52` 表示后面的 JSON body 长度是 `0x52`，也就是 82 字节。

## 5. JSON Envelope

所有消息都使用同一个顶层结构：

```json
{
  "version": "1.0",
  "type": "private_msg",
  "request_id": "b8c9e0f5a3c14a01a86f33d2d8e5a987",
  "timestamp": "2026-05-19T12:00:00Z",
  "sender": "alice",
  "receiver": "bob",
  "group_id": null,
  "payload": {
    "content": "你好"
  },
  "meta": {}
}
```

字段含义：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `version` | string | 否 | 协议版本，默认 `1.0` |
| `type` | string | 是 | 消息类型，相当于本协议里的业务路由 |
| `request_id` | string | 否 | 请求唯一 ID，用于请求/响应关联和日志追踪 |
| `timestamp` | string | 否 | UTC ISO-8601 时间 |
| `sender` | string/null | 否 | 发送方用户名 |
| `receiver` | string/null | 否 | 接收方用户名，私聊和文件私发使用 |
| `group_id` | string/null | 否 | 群组 ID，群聊和群文件使用 |
| `payload` | object | 否 | 业务数据，具体内容由 `type` 决定 |
| `meta` | object | 否 | 扩展元数据，如客户端版本、重试次数、trace 信息 |

`payload` 是本协议的主要扩展点。顶层 Envelope 尽量稳定，后续新增 AI、内容审核、文件传输时，优先扩展 `type` 和 `payload`。

## 6. 消息类型

| 类型 | 说明 | 典型 payload |
| --- | --- | --- |
| `register` | 用户注册 | `{"username":"alice","password":"***"}` |
| `login` | 用户登录 | `{"username":"alice","password":"***"}` |
| `logout` | 用户退出 | `{}` |
| `private_msg` | 一对一私聊 | `{"content":"hello"}` |
| `group_msg` | 群聊消息 | `{"content":"hello group"}` |
| `create_group` | 创建群组 | `{"name":"network-class"}` |
| `join_group` | 加入群组 | `{"group_id":"g1"}` |
| `leave_group` | 退出群组 | `{"group_id":"g1"}` |
| `heartbeat` | 心跳包 | `{"seq":1}` |
| `user_status` | 用户状态更新 | `{"username":"alice","status":"online"}` |
| `history_request` | 请求历史消息 | `{"chat_type":"private","peer":"bob","limit":50}` |
| `history_response` | 返回历史消息 | `{"messages":[]}` |
| `file_start` | 文件传输开始 | `{"filename":"test.pdf","filesize":1024,"file_id":"file-1"}` |
| `file_chunk` | 文件分片 | `{"file_id":"file-1","offset":0,"data":"base64..."}` |
| `file_end` | 文件传输结束 | `{"file_id":"file-1","status":"finished"}` |
| `ai_request` | AI 请求 | `{"prompt":"解释 TCP 和 UDP 的区别"}` |
| `ai_response` | AI 回复 | `{"content":"..."}` |
| `moderation_warning` | 内容审核警告 | `{"reason":"包含违规内容"}` |
| `error` | 错误响应 | `{"error_code":"auth_failed","message":"登录失败","detail":{}}` |

## 7. 错误格式

错误统一使用 `type = "error"`：

```json
{
  "version": "1.0",
  "type": "error",
  "request_id": "same-as-request",
  "timestamp": "2026-05-19T12:00:00Z",
  "sender": "server",
  "receiver": null,
  "group_id": null,
  "payload": {
    "error_code": "auth_failed",
    "message": "用户名或密码错误",
    "detail": {}
  },
  "meta": {}
}
```

推荐错误码：

| 错误码 | 说明 |
| --- | --- |
| `invalid_json` | JSON 解析失败 |
| `invalid_frame` | frame 长度非法 |
| `invalid_message_type` | 未知消息类型 |
| `missing_field` | 缺少必要字段 |
| `invalid_field` | 字段类型或取值非法 |
| `auth_failed` | 注册或登录认证失败 |
| `not_found` | 用户、群组、消息不存在 |
| `conflict` | 重复注册、重复入群等冲突 |
| `server_error` | 服务端内部错误 |

## 8. Python 使用方法

### 8.1 构造并发送消息

```python
from common.protocol import MessageType, ProtocolMessage, encode_frame

message = ProtocolMessage(
    type=MessageType.PRIVATE_MSG,
    sender="alice",
    receiver="bob",
    payload={"content": "你好"},
)

frame = encode_frame(message)
sock.sendall(frame)
```

`encode_frame()` 做了两件事：

1. 把 `ProtocolMessage` 转成 JSON UTF-8 bytes。
2. 在前面加 4 字节 body 长度。

### 8.2 接收并解析消息

```python
from common.protocol import decode_frames

buffer = b""

while True:
    chunk = sock.recv(4096)
    if not chunk:
        break

    buffer += chunk
    messages, buffer = decode_frames(buffer)

    for message in messages:
        handle_message(message)
```

`decode_frames()` 会自动处理：

- 一个 `recv()` 收到半条消息。
- 一个 `recv()` 收到多条消息。
- 多次 `recv()` 才拼出一条完整消息。

### 8.3 服务端路由示例

```python
from common.protocol import MessageType

def handle_message(message):
    if message.type == MessageType.LOGIN:
        handle_login(message)
    elif message.type == MessageType.PRIVATE_MSG:
        handle_private_message(message)
    elif message.type == MessageType.HEARTBEAT:
        handle_heartbeat(message)
    else:
        handle_unknown(message)
```

这里的 `message.type` 类似 HTTP 中的 `POST /login`、`GET /history`，用于决定交给哪个业务处理函数。

## 9. 数据在项目中的流动

一条私聊消息在项目中的流动如下：

```text
客户端 UI 输入
-> 构造 ProtocolMessage(type="private_msg")
-> encode_frame()
-> socket.sendall()
-> 服务端 socket.recv()
-> decode_frames()
-> message_router 根据 type 分发
-> database.save_message() 保存历史
-> 转发给 Bob 的 socket
-> Bob 客户端 decode_frames()
-> UI 显示消息
```

一条登录消息的流动如下：

```text
客户端输入用户名密码
-> ProtocolMessage(type="login", payload={...})
-> TCP 发送
-> 服务端解析
-> database.authenticate_user()
-> 成功则返回 user_status 或登录成功响应
-> 失败则返回 type="error"
```

## 10. 扩展规范

- 新功能优先新增 `type` 或扩展 `payload` 字段。
- 不要随意改名或删除 Envelope 顶层字段。
- 新增消息类型必须同步更新 `MessageType` 和本文档。
- 文件、AI、审核等扩展能力应复用 `request_id` 追踪请求链路。
- 大文件不要直接放入单个 JSON，应使用 `file_start`、`file_chunk`、`file_end` 分片传输。
- `meta` 只放调试、追踪、客户端版本等元信息，不放主要业务内容。

## 11. 设计总结

FastAPI 项目并不是没有协议，而是直接使用了 HTTP。HTTP 已经帮开发者解决了消息边界、请求头、body 长度、路由、状态码等问题。

本项目为了练习计算机网络课程内容，直接使用 TCP socket。TCP 只提供可靠字节流，不提供聊天消息边界，所以我们必须自己定义应用层协议。

本协议的核心是：

```text
应用层 frame = 4 字节长度前缀 + JSON Envelope
```

其中：

- 4 字节长度前缀解决粘包、拆包、半包问题。
- JSON Envelope 表达业务语义。
- `type` 类似 HTTP 的 URL/method，用于业务路由。
- `payload` 类似 HTTP body，用于承载具体业务数据。
- `error` 消息类似 HTTP 错误响应，用于统一表达失败原因。

这套设计让项目既能体现网络分层和 TCP 编程能力，又能保持代码易读、易调试、易扩展。
